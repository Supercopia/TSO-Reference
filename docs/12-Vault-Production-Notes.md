## Unit Package System cluster (4 docs)

The interlocking architecture record for the Unit Package System. The most important cluster in the vault for understanding how the [Unit Hierarchy](02-Unit-Hierarchy.md) actually composes.

### 1. Architecture Fix

Diagnoses what was wrong with the original `UnitNodePackage` self-destructing-reparent pattern and explains the replacement: **packages stay permanent.** Bylined "By #claude_code, with lots of input from me. (Acadee)".

The original `UnitNodePackage` tried to self-destruct at runtime by reparenting its children to the Unit, then deleting itself. This fought Godot at every turn:

- Godot fires `_ready()` bottom-up — children's `_ready()` ran while their parent was still the package, not the Unit.
- `await get_parent().unpacked` did nothing — `unpacked` was a `bool`, not a `Signal`.
- `add_child.call_deferred()` reparents too late — the Unit's child-scan ran before any reparenting.
- StateNodes in packages never registered — `_refresh_special_states()` only scanned direct children.

The fix:

> Packages **never self-destruct.** They stay in the scene tree as permanent group nodes — think of them as labelled folders that happen to also be Nodes. Their children walk up the tree to find the Unit themselves using a shared static helper. The Unit scans recursively when it needs to find parts.
>
> Zero deferred calls. Zero await-on-bool. Zero timing fights with Godot.

The shared helper:

```gdscript
static func find_unit(from: Node) -> Node:
    var n = from.get_parent()
    while n and not n is CharacterBody2D:
        n = n.get_parent()
    return n
```

Three other bugs the rewrite fixed along the way:

- `child = attack_hitbox` backwards assignment in `UnitClass.get_children_things()` (hitbox always null).
- `Stand_or_Range` boolean inverted in `RangeDetectors._ready()` (`_physics_process` had the correct branches; `_ready` was wrong).
- Nudge running every frame in `DamageReceiver2D._physics_process` (moved to one-time call in `_ready()` after 0.1s timer).

The full per-file change list — `UnitNodePackage.gd`, `RangeDetectors.gd`, `damage_receiver_2d.gd`, `HitboxDamageArea2D.gd`, `StateNodeClass.gd`, `UnitClass.gd`, `AttackNode.gd` — is in `_research/vault-extraction.md` § 2.1.

### 2. Developer Guide

The teaching doc. Bylined "Generated by Claude Code, after absorbing AcadeeAlkana's ramblings." Aimed at teammates, modders, future-Claude.

The three rules of writing a package-child script:

1. **Always store `_unit_ref` in `_ready()`** (not `_physics_process`).
2. **Always null-check `_unit_ref`** before using it in `_physics_process`.
3. **Never use `get_parent()`** to reach the Unit — your parent is the package, not the Unit.

Worked example: a "Regeneration" status node, ~20 lines of script. Plus three smaller examples covering "telling the Unit about yourself" (the pattern `_unit_ref.attack_hitbox = self` uses), "reading from the Unit every frame", and the null-safe guard pattern.

Includes a table of what can and can't be inside a package:

| Node type | In a package? |
|---|---|
| `Node`, `Node2D`, `Area2D`, `RayCast2D`, `CollisionShape2D` | Yes |
| `StateNode` | Yes (the recursive scan finds them) |
| `AnimationPlayer` | **No** — keep as direct child of the Unit |
| `Sprite2D` / `UnitGraphics` | **No** — same reason |

Why Node2D specifically (not plain Node): Godot's `NOTIFICATION_TRANSFORM_CHANGED` only propagates through `Node2D` ancestors. A plain `Node` swallows the notification silently; child `Area2D`/`RayCast2D` get stuck at the spawn position.

Full Quick Reference card in `_research/vault-extraction.md` § 2.2.

### 3. Collision Layers

The layer/mask reference table. The convention is now codified in three places in code (the addon's `damage_receiver_2d.gd`, `HitboxDamageArea2D.gd`, `RangeDetectors.gd`); this doc is the design source-of-truth.

| Layer | Decimal | Role | Owner |
|---|---:|---|---|
| 1 | 1 | Ally front-of-unit | `FrontDetector` Area2D (ally) |
| 2 | 2 | Ally hitbox | `HitboxDamageArea2D` (ally) |
| 3 | 4 | Ally hurtbox | `DamageReceiver2D` (ally) |
| 4 | 8 | Reserved | (future: Ally vision / Nurse Slime healing range) |
| 5 | 16 | Enemy front-of-unit | `FrontDetector` Area2D (enemy) |
| 6 | 32 | Enemy hurtbox | `DamageReceiver2D` (enemy) |
| 7 | 64 | Enemy hitbox | `HitboxDamageArea2D` (enemy) |
| 8 | 128 | Reserved | (future: Enemy vision) |

The hitbox/hurtbox layers are **deliberately not symmetric** across teams (ally hitbox=2, hurtbox=3 vs. enemy hitbox=7, hurtbox=6) so that masks set in one direction can't accidentally double-resolve in the other.

`front_of_unit` is a separate Area2D inside the Hurtbox package — attacks pass through it, only range rays interact with it. Set at runtime by `DamageReceiver2D._ready()` after the 0.1 s team-direction timer.

Mask table (who detects whom), all layer/mask code citations, and the full per-file-assignment list in `_research/vault-extraction.md` § 2.3.

## Unit Effects (md + canvas)

Near-complete design spec for the trigger-event / status-node architecture that should eventually drive Strives, abilities, and TBC-style mechanics. Begins: "And what can they do?"

### Trigger Events / Receivers

The Unit emits Trigger Events when it does **literally anything**. Status Nodes attached to the Unit subscribe via signal. The emitted events:

- **When Alive** — every frame
- **When Knockbacked** — once at KB; sub-options for "at X% of KB duration remaining"
- **When Moving** / **When Idle** — every frame in this state, with "frames in a row" tracking
- **When Fighting** vs **When Gathering** — depending on Opponent vs Non-Opponent in attack range
- **When Targeting** — every frame while navigating to a specific target
- **When Busy with an Action** — attacks, special actions, KB, jumps
- **When Attacking** — every frame of foreswing / hit-duration / backswing, with "% of Y remaining" sub-options
- **When Unit Has A Special State** / **When Unit Has A Status Node** — observer pattern over names
- **Upon Dying** — once, frame health < Minimum
- **When Taking A Hit** / **When Hitting** — single-frame events
- **When Seeing Entity** — via Standing/Attacking Range raycast or dedicated Detection Range RayCast2D
- **When (Total Duration of any above state)** — lifetime accumulator

Two design examples called out:

- **Nurse Slime:** When Seeing Entity (Ally Unit), When Entity is Knocked Back → "Heal them" (Special Action).
- **Prospector Enemy:** When Seeing Entity (Enemy Unit), When SELF is Idle → "Buff them with Gold HP" (Special Action).

### Status Node effects

Status Nodes receive Trigger Signals and do things — modify the Unit's stats, change movement type, spawn new Status Nodes, give passive/active abilities, etc. The doc enumerates ~12 effect categories from "Reduce Duration" through "Spawn Nodes" (VFX, Bandaids, totems).

### Stat counts

The doc gives a precise stat count per Unit:

- **Health (3):** Current, Maximum, Minimum (death threshold)
- **Move Speed (2):** Current, Base (for % buffs)
- **Attack Cooldown (1):** time between attacks
- **Attack Speed (4):** Foreswing, Midswing, Backswing, Entire Animation Multiplier
- **Current Animation being Played** — forceable
- **Range** — Attacking, Standing, Seeing
- **Attack Hitbox Distance** — Length, Offset
- **Able to Act / Move / Attack** — three boolean gates

### TBC effects to recreate

The doc spends ~70% of its length recreating Battle Cats Talents/abilities by mapping each to the Trigger / Status Node system. Highlights:

- **Knockbacks** — three preset levels with frame counts and distances (Cat Cannon, Damage-Based, Boss Shockwave)
- **Strong/Resistant/Insanely Tough vs Trait** — damage multipliers (1.8x, 1/5, 1/7)
- **Critical Hit / Savage Blow / Metal / Barrier / Aku Shield / Guard Cracker** — chance-based or threshold-based
- **Toxic / Warp / Curse / Push / Weaken / Freeze / Slow** — debuffs with specific frame counts and durations
- **Soulstrike** — phases through invulnerability, leaves delayed Soul Damage
- **Wave / Surge / Explosion / Mini variants** — spawn-scene pattern with damage % nerfs

Full enumeration in `_research/vault-extraction.md` § 3.1.

### Unit Effects CANVAS

The operational flowchart for what happens each tick. Twelve text nodes, no labeled edges. The diagram in prose:

```
Normal Unit Functions → do normal things
        │
        ▼ (every tick)
emit Trigger Signals to connected Effect Nodes
        │
        ▼
for each connected Effect Node: "Is Trigger Signal Accepted?"
   ├── No  → back to Normal Unit Functions
   └── Yes → Run Effect Node functions
                │
                ▼
            "Stopping Condition Met?"
               ├── No  → keep running (next tick)
               └── Yes → call Expire() on the Effect Node, despawn
                              │
                              ▼
                         Return to Normal Unit Functions
```

Full node and edge tables in `_research/vault-extraction.md` § 3.2.

## Macro Progression Matters

The leveling / strives / training spec. Opens: "We do want them for this kinda thing."

### Leveling

Each Level raises Unit stats by ~16% (additive). Per-Level milestones:

- **Level 1** — Starts with 2 Strive Slots
- **Level 6, 11, 16, 21, 26** — Strive Capacity +1
- **Level 4, 14, 24** — Training Clock spendable (1, 2, 3 times)
- **Level 8, 18, 28** — Training Points +1
- **Level 10** — natural Extra Form (EX) unlocks if Unit has one
- **Level 20** — natural Deluxe Form (DX) unlocks if Unit has one
- **Level 26** — Maximum Strives Count +1; **"Past Level 26, things get much more expensive. It's okay to stop Leveling here!"**
- **Level 33** — MAX LEVEL, +37% → final total 614%

Full level-by-level table in `_research/vault-extraction.md` § 5.

### Strive keywords

- **Trainable** — Training Point spent → Strive costs 0 SP in stages
- **Priceless** — cannot be activated with SP
- **Overtime** — TP spent yields a free copy of the Strive instead of reducing price
- **Hardworking** — cannot equip unless TP spent; more TP makes it free
- **X Form Only** — Form-locked; equip on other Forms gives a +1% to all stats consolation
- **The Bottom 1%** — "available to everyone... But, like. Why" — +1% all stats consolation Strive

### Training Clocks

First-time stage rewards. Once spent, gone — but there's enough in the game to max everyone if you want. Each Clock = 1 Training Point.

### Balancing intent

The doc spends most of its length on why Trainable + generic stat increases is the right combination: it invites min-maxing without creating broken active-cooldown abuse, makes generic stat Strives feel valuable when they're free, and acknowledges that filling all 7 Strive Slots is intentionally hard to do via purchase alone.

- Slow and steady wins the race
- Code things the clean way, build things to last
